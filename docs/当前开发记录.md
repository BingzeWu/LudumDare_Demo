# 当前开发记录

本文档用于记录当前涂鸦跳跃原型的已完成改动、当前有效方案和下次继续开发时的切入点。

## 1. 当前项目状态

- 项目方向已经切换为竖屏涂鸦跳跃原型
- 当前项目分辨率为 `720 x 1000`
- 主玩法场景仍然是程序生成平台
- 已新增独立的平台测试场景，方便在编辑器里手动摆放平台测试效果

当前关键入口：

- 主玩法场景：`res://scenes/doodle_jump/doodle_jump_game.tscn`
- 平台测试场景：`res://scenes/doodle_jump/platform_test_scene.tscn`

## 3. 最近这轮完成的关键改动

### 3.1 分辨率联动

在 `jumper_game.gd` 中加入了运行时同步逻辑：

- 可自动把 `world_width` 同步到当前视口宽度
- 可自动把 `base_platform_y` 同步到当前视口高度
- 初始平台位置和玩家出生位置会跟随分辨率变化调整

当前相关参数：

- `auto_sync_world_width_to_viewport`
- `auto_sync_base_platform_y_to_viewport`
- `base_platform_bottom_margin`
- `player_spawn_offset_from_base_platform`

### 3.2 平台测试场景

新增文件：

- `res://scenes/doodle_jump/platform_test_scene.tscn`
- `res://scripts/doodle_jump/platform_test_scene.gd`

用途：

- 不走程序生成
- 在编辑器中把不同平台拖到 `Platforms` 节点下即可测试
- 适合验证单次平台、移动平台、坚固平台的行为

### 3.3 平台默认颜色修正

之前平台场景里的 `Polygon2D.color` 会被 `BasePlatform.tint` 覆盖，导致测试场景里所有平台都显示成绿色。

已修复为各平台场景显式写入自己的 `tint`：

- 普通平台：绿色
- 单次平台：黄色
- 移动平台：蓝色
- 坚固平台：紫色

### 3.4 左右穿屏修正

之前玩家左右穿屏逻辑只会在“落地反弹后”执行，导致空中横移时不能正确穿屏。

当前处理方式：

- 玩家每帧 `move_and_slide()` 后都会执行 `_wrap_horizontally()`

## 4. 当前移动平台的有效方案

这一条很重要，下次继续开发时不要回退。

当前移动平台采用的方案是：

1. 使用平台左右边缘和左右墙坐标判断是否碰壁
2. 碰到左墙时只切换方向为向右
3. 碰到右墙时只切换方向为向左
4. 不再做“碰壁后强行把平台位置钉到边界”的位置修正

原因：

- 之前尝试过在碰壁时把平台中心直接强制设置到边界内侧
- 实际测试结果表明这种写法会让平台在一侧看起来卡住
- 用户已经手动验证：去掉那两行位置修正后，平台移动效果恢复正常

因此当前保留的版本是“只切换方向，不修正位置”。

涉及文件：

- `res://scripts/doodle_jump/platforms/moving_platform.gd`

## 5. 当前测试场景的使用方法

打开：

- `res://scenes/doodle_jump/platform_test_scene.tscn`

用法：

1. 在编辑器里选中 `Platforms`
2. 把不同平台场景拖进去，或者复制已有样例
3. 调整平台位置、尺寸、数量
4. 运行场景进行测试
5. 按 `Enter` 重置测试状态

当前测试场景特性：

- 左右边界有可视化参考线
- 镜头默认不跟随玩家横向位置
- 镜头默认跟随玩家纵向位置

说明：

- 之所以关闭横向跟随，是因为测试左右穿屏时，如果镜头也一起横向瞬移，视觉上会很难判断穿屏是否正常

## 6. 当前已知规则和限制

### 6.1 平台生成器

当前程序生成平台时：

- 只严格约束纵向间距
- 横向只限制不贴边
- 没有做“同层平台数量限制”
- 没有做“相邻平台最小水平距离限制”
- 没有做“附近几层避免连续移动平台”的约束

这意味着主玩法场景里仍可能出现：

- 平台看起来在同一层过于密集
- 多个移动平台挤在接近高度

这部分还没有进一步收敛。

### 6.2 文档同步

已有文档：

- `docs/doodle_jump_breakdown.md`
- `docs/doodle_jump_logic_chain.md`

它们记录了主玩法结构和逻辑主链，但还没有完整补入“平台测试场景”和“移动平台当前最终方案”的所有细节。

因此这份文档作为当前开发续接记录优先参考。

## 7. 下次继续开发建议起点

如果下次继续开发，建议按这个顺序推进：

1. 先在 `platform_test_scene.tscn` 里验证不同平台行为是否符合预期
2. 再回到 `doodle_jump_game.tscn` 调整平台生成规则
3. 优先补下面两个生成器约束：
   - 相邻平台最小水平距离
   - 附近几层移动平台数量限制
4. 等平台手感稳定后，再考虑加怪物、陷阱或更复杂的道具

## 8. 本次建议记住的结论

- 分辨率改变后，主玩法里的逻辑宽度和出生高度需要跟着同步
- 平台测试最好在独立测试场景里做，不要直接在主玩法生成器里硬测
- 移动平台当前有效方案是“边缘碰壁后只切方向，不做位置强制修正”
- 左右穿屏测试时，镜头不要跟随玩家横向位置
