# 简易涂鸦跳跃逻辑链路说明

本文档说明当前竖屏版 `DoodleJumpGame` 原型里，一条完整的玩法链路是如何从规则设计落到场景节点、脚本职责和运行时行为上的。

## 1. 核心玩法目标

当前原型优先保证“能稳定玩一局”，所以所有实现都围绕下面这条主链展开：

1. 玩家左右移动，控制落点
2. 玩家在重力作用下下落
3. 玩家从上方踩中平台后自动反弹
4. 相机只记录向上的推进，不跟着回落
5. 相机上方持续生成平台和道具
6. 掉到镜头下方的旧内容被回收到对象池
7. 玩家掉出镜头底部后结束本局

这意味着最关键的不是资源，而是这些运行时关系：

- `Player` 负责位移、重力、自动弹跳、道具状态
- `Platforms` 负责提供可落脚表面
- `Pickups` 负责临时改变跳跃状态
- `Camera2D` 负责定义“当前有效可玩区域”
- `DoodleJumpGame` 负责生成、回收、计分、失败判定和界面刷新

## 2. 竖屏场景结构

主场景文件：

- `res://scenes/doodle_jump/doodle_jump_game.tscn`

当前采用竖屏视口 `720 x 1280`，主要节点作用如下：

- `DoodleJumpGame`
  作用：主控制器，挂载 `jumper_game.gd`

- `BackgroundLayer/Background`
  作用：固定背景色
  说明：放在 `CanvasLayer` 下，不受相机移动影响

- `Platforms`
  作用：活动平台容器
  说明：所有平台实例都挂在这里，便于统一管理和回收

- `Pickups`
  作用：活动道具容器
  说明：所有道具实例都挂在这里，便于统一管理和回收

- `Player`
  作用：玩家角色
  说明：负责输入、重力、自动起跳、穿屏和道具状态

- `Camera2D`
  作用：主镜头
  说明：只会继续上移，不会因玩家回落而向下追

- `UILayer/Hud`
  作用：顶部状态栏
  说明：显示高度与提示文本

- `UILayer/GameOverPanel`
  作用：结算面板
  说明：显示最终高度，并提供重开和返回菜单入口

- `PauseMenuLayer`
  作用：复用模板暂停菜单
  说明：暂停期间主逻辑停止推进

## 3. 开局初始化链路

实现位置：

- `res://scripts/doodle_jump/jumper_game.gd`

关键函数：

- `_ready()`
- `_start_run()`

执行过程如下：

1. 进入场景后执行 `_ready()`
2. 初始化随机数生成器
3. 把 `world_width` 传给 `Player`，保证穿屏边界与关卡宽度一致
4. 连接玩家落地信号、重开按钮、返回菜单按钮
5. 调用 `_start_run()` 启动一局新游戏

`_start_run()` 负责完成这些重置：

1. 调整背景大小到当前视口
2. 回收上一局仍在活动中的平台和道具
3. 清空失败状态并隐藏结算面板
4. 把玩家放到竖屏下半区的初始位置
5. 把相机放到视口中线，形成稳定的初始观察区
6. 记录当前最高点、最高生成位置和分数初值
7. 先生成一个更宽的出生平台
8. 循环调用 `_spawn_next_platform()`，把相机上方缓冲区提前铺满

开局可玩的原因，就是这一步已经同时准备好了“玩家、出生平台、相机、分数、上方平台带”。

## 4. 玩家移动与自动起跳链路

实现位置：

- `res://scripts/doodle_jump/jumper_player.gd`

关键函数：

- `_physics_process(delta)`
- `_wrap_horizontally()`
- `_find_floor_platform()`

### 4.1 左右移动

1. 读取 `move_left` 和 `move_right`
2. 通过 `Input.get_axis()` 得到 `-1` 到 `1` 的方向值
3. 使用 `move_toward()` 把当前水平速度渐变到目标速度
4. 再通过 `move_and_slide()` 执行实际位移

这样做的效果是：

- 输入手感不会过硬
- 玩家控制的重点是“调整落点”，不是爆发冲刺

### 4.2 重力与自动反弹

1. 若当前没有火箭效果，每帧给 `velocity.y` 叠加重力
2. 用 `max_fall_speed` 限制最大下落速度
3. 记录当前是否处于下落状态
4. 调用 `move_and_slide()` 更新物理碰撞
5. 如果之前在下落且现在检测到落地，则把 `velocity.y` 直接改成 `jump_velocity`
6. 若落地对象是平台，发出 `bounced(platform)` 信号，让平台自行决定后续反馈

这里通过“先下落再判定落地”的顺序，保证只有真正从上方踩中平台时才会触发反弹。

### 4.3 左右穿屏

1. 每帧移动结束后检查 `global_position.x`
2. 小于左边界缓冲值时，直接传送到右侧
3. 大于右边界缓冲值时，直接传送到左侧

这条链路让竖屏窄场地仍然保持较高的横向容错率。

## 5. 相机推进链路

实现位置：

- `res://scripts/doodle_jump/jumper_game.gd`

关键函数：

- `_update_camera()`

执行过程：

1. 用玩家当前位置减去 `camera_lead` 计算目标镜头高度
2. 使用 `min(camera.position.y, target_y)` 更新相机 Y

因为 Godot 的 2D 坐标里，Y 越小代表位置越高，所以这里取更小值，就能实现“只向上跟随、不向下回退”。

竖屏模式里把 `camera_lead` 调大后，相机可以更早暴露上方平台，减少玩家被视野外落点坑到的情况。

## 6. 平台生成链路

实现位置：

- `res://scripts/doodle_jump/jumper_game.gd`

关键函数：

- `_spawn_platforms_if_needed()`
- `_spawn_next_platform()`
- `_spawn_platform()`
- `_roll_platform_type()`

执行过程：

1. 每帧根据相机位置计算可视区域顶部
2. 检查当前“已生成的最高平台”是否高于顶部缓冲区
3. 如果不够高，就继续生成下一批平台
4. 生成单个平台时依次完成：
   - 随机一个垂直间距
   - 随机一个平台宽度
   - 在左右安全边界之间随机一个横坐标
   - 按概率决定平台类型
   - 从对象池取出对应平台实例
   - 调用 `activate()` 刷新位置、尺寸、颜色和碰撞
   - 如果是移动平台，再调用 `setup_movement()` 配置移动范围和速度

这一步把“无限上升”转成了运行时的内容供给系统。

### 6.1 平台类型链路

- 恒定平台：普通落脚点，不附带额外行为
- 单次平台：收到玩家踩踏回调后请求回收
- 移动平台：在自己的 `_physics_process()` 中左右往返运动
- 坚固平台：关闭单向穿透，强化阻挡感

主控制器只决定“生成什么”，具体平台自己决定“踩到后发生什么”。

## 7. 道具生成与生效链路

实现位置：

- `res://scripts/doodle_jump/jumper_game.gd`
- `res://scripts/doodle_jump/pickups/`

关键函数：

- `_try_spawn_pickup()`
- `_spawn_pickup()`
- `_on_pickup_consumed()`

执行过程：

1. 新平台生成后，主控制器判断该平台是否允许挂道具
2. 当前规则下，坚固平台和移动平台不会挂道具，避免节奏过乱
3. 如果命中生成概率，就从对象池取出弹簧或火箭
4. 调用 `activate()` 设置位置、半径、颜色和碰撞
5. 玩家进入道具碰撞范围后，道具脚本触发效果
6. 弹簧调用 `player.trigger_spring()`
7. 火箭调用 `player.trigger_rocket()`，让玩家在一段时间内持续上升并关闭平台碰撞
8. 道具通过 `consumed` 信号请求主控制器回收自己

这样一来，道具系统既能插入主玩法，又不会把复杂状态散落到多个地方。

## 8. 对象池回收链路

实现位置：

- `res://scripts/doodle_jump/jumper_game.gd`
- `res://scripts/doodle_jump/platforms/base_platform.gd`
- `res://scripts/doodle_jump/pickups/base_pickup.gd`

这部分是当前性能稳定的关键。

### 8.1 为什么不用频繁创建和销毁

平台和道具会随着玩家持续上升而不断出现、消失。如果每次都 `instantiate()` 再 `queue_free()`，节点数量虽然能回落，但运行时会频繁分配和释放对象，容易带来卡顿和“看起来像泄漏”的内存抖动。

### 8.2 当前回收方式

1. 首次需要某个平台或道具时，主控制器实例化一个节点
2. 节点加入容器后不会再被销毁，而是进入对应类型的池
3. 再次需要同类节点时，优先从池里取
4. 节点离开可玩区域或被消耗后，不销毁，而是执行 `deactivate()`
5. `deactivate()` 会隐藏节点、关闭碰撞、重置状态
6. 节点回到池数组中等待下一次 `activate()`

### 8.3 平台回收触发点

1. `_cleanup_platforms()` 计算相机底部
2. 遍历活动平台数组
3. 低于镜头底部加上回收边距的平台，调用 `_recycle_platform()`
4. 单次平台也可以在被踩踏后主动发出 `recycle_requested`，提前进入池

### 8.4 道具回收触发点

1. `_cleanup_pickups()` 计算相机底部
2. 遍历活动道具数组
3. 低于镜头底部加上回收边距的道具，调用 `_recycle_pickup()`
4. 被玩家吃到的道具会通过 `consumed` 信号回收到池

对象池和“上方生成、下方回收”共同构成了当前游戏的内容循环闭环。

## 9. 计分链路

实现位置：

- `res://scripts/doodle_jump/jumper_game.gd`

关键函数：

- `_update_score()`
- `_refresh_score_labels()`

执行过程：

1. 持续记录玩家到达过的最高位置 `_highest_player_y`
2. 用 `base_platform_y - _highest_player_y` 计算本局累计爬升高度
3. 把结果刷新到状态栏和结算面板

这种计分方式的优点是：

- 不依赖平台数量
- 不依赖时间
- 直接反映玩家在本局里真正爬了多高

## 10. 失败与重开链路

实现位置：

- `res://scripts/doodle_jump/jumper_game.gd`

关键函数：

- `_check_game_over()`
- `_restart()`
- `_back_to_menu()`

执行过程：

1. 每帧根据相机位置计算当前镜头底部
2. 如果玩家掉到镜头底部以下，并超过 `game_over_margin`
3. 将 `_is_game_over` 设为 `true`
4. 调用 `player.disable()` 停止角色物理更新
5. 刷新结算高度文本
6. 显示结算面板
7. 玩家可以按 `Enter` 或点击按钮重新开局，也可以返回主菜单

这一步让“一局结束”成为明确的状态切换，而不是让玩家继续在场景里无意义地下坠。

## 11. 一条完整的运行时主链

把整局游戏压缩成一条执行顺序，可以看成：

1. 场景加载
2. `_ready()` 初始化依赖
3. `_start_run()` 生成开局状态
4. `Player` 每帧更新输入、重力、自动弹跳、穿屏
5. `DoodleJumpGame` 每帧更新相机、分数、平台生成、道具生成和回收
6. 玩家通过平台和道具持续向上推进
7. 玩家失误跌出镜头后触发失败
8. 界面显示结算
9. 玩家选择重开或返回菜单

这就是当前原型从“游戏规则”到“功能实现”的完整链路。
